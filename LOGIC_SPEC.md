# Логіка роботи Academic Assistant — повна специфікація

---

## ВХІДНІ ДАНІ (що може надати клієнт)

| Поле | Тип | Опис |
|---|---|---|
| Шаблон замовлення | текст | тема, тип роботи, сторінки, мова, дедлайн |
| Методичка | PDF | вимоги кафедри до структури та оформлення |
| Коментар клієнта | текст + фото | довільний текст, може містити план, вимоги, побажання |
| Готовий план | текст | клієнт вставив власний план вручну |

---

## ЕТАП 1: АНАЛІЗ ВХІДНИХ ДАНИХ

### 1.1 Аналіз шаблону (завжди, Haiku)
Витягує з тексту шаблону:
- `type` — тип роботи (курсова, дипломна тощо)
- `topic` — тема
- `subject` — галузь/предмет
- `pages` — обсяг (може бути діапазон "30-40")
- `language` — мова роботи
- `deadline`, `uniqueness`, `extras`

**Правило сторінок:** якщо діапазон "30-40" → беремо середнє → 35.

---

### 1.2 Читання методички (тільки якщо завантажено PDF, Gemini)
Витягує структурну та оформлювальну інформацію.

**Що має витягуватись:**

```
СТРУКТУРА:
- chaptersCount         — к-сть розділів (null якщо не вказано)
- subsectionsPerChapter — к-сть підрозділів у розділі (null якщо не вказано)
- hasChapterConclusions — true ТІЛЬКИ якщо явно вимагається
- chapterTypes          — ["theory", "analysis", "recommendations"]
- exampleTOC            — зразок змісту зі студентської роботи (null якщо немає)

СТОРІНКИ:
- totalPages            — загальний обсяг (число)
- introPages            — обсяг вступу (null якщо не вказано)
- conclusionsPages      — обсяг висновків (null якщо не вказано)

ВСТУП:
- introComponents       — перелік елементів вступу (null → стандартні 7)

ВИМОГИ ДО ТЕКСТУ:
- theoryRequirements           — що має бути в теоретичному розділі
- analysisRequirements         — що має бути в аналітичному розділі
- chapterConclusionRequirements — вимоги до висновків до розділу
- conclusionsRequirements      — вимоги до загальних висновків
- otherRequirements            — інші вимоги

ОФОРМЛЕННЯ:
- sourcesMinCount   — мінімум джерел
- sourcesStyle      — стиль (APA / ДСТУ 8302:2015 / MLA тощо)
- sourcesOrder      — alphabetical / citation_order
- sourcesGrouping   — правила групування (мова, тип)
- citationStyle     — як цитувати в тексті
- formatting        — шрифт, поля, інтервали, відступи тощо
```

**Важливо щодо exampleTOC:**
- Шукати ТІЛЬКИ розділ зі словами "зразок змісту", "зразок плану", "приклад змісту"
- Зміст самої методички (на початку PDF) — ігнорувати
- Якщо не знайдено → null (не вигадувати)

---

### 1.3 Аналіз коментаря клієнта (якщо є коментар або фото, Haiku)

Коментар може містити будь-що: план, вимоги до змісту, стиль, джерела, оформлення.

**Що витягується (4 типи підказок + план з фото):**

```
planHints          — підказки для СТРУКТУРИ ПЛАНУ
                     (к-сть розділів, назви, висновки до розділів тощо)

textStructureHints — підказки для СТРУКТУРИ ТЕКСТУ
                     (що має бути у вступі, висновках, акценти)

writingHints       — підказки для СТИЛЮ ТА ЗМІСТУ написання
                     (термінологія, підходи, що підкреслити)

sourcesHints       — підказки для ДЖЕРЕЛ ТА ОФОРМЛЕННЯ
                     (к-сть, мова, стиль цитування, автори)

photoTOC           — якщо на фото є готовий план/зміст —
                     скопіювати текст дослівно, інакше null
```

## ЕТАП 2: ПОБУДОВА РОБОЧОГО КОНФІГУ (WorkConfig)

Після аналізу всіх вхідних даних — збирається єдиний об'єкт.
Кожне поле заповнюється з першого доступного джерела за пріоритетом.

### 2.1 Загальний обсяг
```
totalPages = середнє(info.pages)
  Приклад: "30-40" → 35, "60" → 60
```

### 2.2 Структура (пріоритет)
```
Джерело 1 (найвищий): план клієнта текстом (clientPlan)
Джерело 2:            план клієнта з фото (photoTOC)
Джерело 3:            коментар містить "Розділ X" (planHints)
Джерело 4:            методичка (methodInfo)
Джерело 5 (дефолт):   автоматично

Що береться:
- chaptersCount     → з джерела або: totalPages >= 40 ? 3 : 2
- subsPerChapter[]  → масив по кожному розділу, наприклад [4, 2, 3]
                      (береться з парсингу плану або з методички)
- hasChapterConclusions → true/false
- chapterTypes      → ["theory","analysis","recommendations"] (перші N)
```

### 2.3 Сторінки
```
introPages:
  1. методичка вказала явно (introPages з methodInfo)    → брати
  2. textStructureHints з коментаря містить обсяг вступу → парсити
  3. дефолт → 2 стор

conclusionsPages:
  1. методичка вказала явно (conclusionsPages з methodInfo) → брати
  2. textStructureHints містить обсяг висновків            → парсити
  3. дефолт → totalPages > 40 ? 3 : 2

chapConclusionPages:
  завжди 1 стор (без винятків)

pagesForMainPart = totalPages - introPages - conclusionsPages
                   - (hasChapterConclusions ? chaptersCount : 0)

pagesPerSubsection = round(pagesForMainPart / totalSubsCount)
  де totalSubsCount = sum(subsPerChapter[])
  мінімум: 3 стор на підрозділ
```

### 2.4 Вимоги до тексту
```
introComponents:
  методичка || стандартні 7 елементів:
  ["актуальність теми", "мета дослідження", "завдання дослідження",
   "об'єкт дослідження", "предмет дослідження",
   "методи дослідження", "структура роботи"]

theoryRequirements:     methodInfo.theoryRequirements || ""
analysisRequirements:   methodInfo.analysisRequirements || ""
conclusionsRequirements:  methodInfo.conclusionsRequirements || ""
chapterConclusionRequirements: methodInfo.chapterConclusionRequirements || ""
otherRequirements:      methodInfo.otherRequirements || ""

writingHints:           commentAnalysis.writingHints || ""
textStructureHints:     commentAnalysis.textStructureHints || ""
sourcesHints:           commentAnalysis.sourcesHints || ""
```

### 2.5 Оформлення джерел
```
sourcesStyle:    methodInfo.sourcesStyle || "ДСТУ 8302:2015"
sourcesOrder:    methodInfo.sourcesOrder || "alphabetical"
sourcesGrouping: methodInfo.sourcesGrouping || ""
citationStyle:   methodInfo.citationStyle || "(Автор, рік)"
sourcesMinCount: methodInfo.sourcesMinCount || (totalPages >= 40 ? 40 : 20)
```

## ЕТАП 3: ГЕНЕРАЦІЯ ПЛАНУ

Використовує WorkConfig. Логіка залежить від того звідки взята структура.

### Гілка A — clientPlan (текст від клієнта)
- JS парсить текст: рядки `РОЗДІЛ X` або `X.` → розділи, рядки `X.Y` → підрозділи
- `Висновки до розділу` → hasChapterConclusions = true
- Все інше (назва розділу на 2-му рядку, порожні рядки) → ігнорується
- План будується БЕЗ AI-запиту, одразу

### Гілка B — photoTOC (план з фото)
- AI вже розпізнав план під час аналізу (етап 1.3)
- Рахуємо підрозділи regex `\d+\.\d+` — тільки нумеровані
- Haiku: "збережи структуру (к-сть розділів та підрозділів), придумай нові назви під тему"

### Гілка C — planHints містить структуру (коментар з "Розділ X")
- Рахуємо підрозділи regex `\d+\.\d+`
- Haiku: "збережи структуру, придумай нові назви"

### Гілка D — методичка (methodInfo)
**D1 — є exampleTOC:**
- JS рахує підрозділи per розділ: тільки рядки `\d+\.\d+` (нумеровані)
- Результат: subsPerChapter = [4, 2, 3] наприклад
- Haiku отримує: "Розділ 1: 4 підрозділи, Розділ 2: 2 підрозділи..." — не змінювати к-сть

**D2 — немає exampleTOC, але є chaptersCount і subsectionsPerChapter:**
- Haiku отримує числа і будує план

### Гілка E — нічого немає (дефолт)
- JS будує базову структуру: 2 або 3 розділи, по 3 підрозділи
- Haiku придумує тільки назви підрозділів

### Пріоритет між гілками:
```
A → B → C → D → E
Якщо A спрацювала — B,C,D,E не запускаються
```

## ЕТАП 4: ГЕНЕРАЦІЯ ТЕКСТУ

Для кожного підрозділу по черзі. Використовує WorkConfig.

### Що передається в кожен промпт:

**Завжди:**
```
- тема, тип роботи, галузь
- план (список всіх підрозділів) — щоб не повторюватись
- контекст: перші 500 символів двох попередніх підрозділів
- к-сть абзаців: pages × 3.5 (мінімум 3)
- ліміт токенів: pages × 3000 (мін 8000, макс 60000)
- writingHints з коментаря (якщо є)
```

**Для ВСТУПУ додатково:**
```
- introComponents (з методички або стандартні 7)
- к-сть завдань = к-сть підрозділів основної частини (макс 5 або 8 якщо >40 стор)
- textStructureHints (якщо є)
- otherRequirements з методички (якщо є)
```

**Для ВИСНОВКІВ додатково:**
```
- к-сть абзаців: totalPages > 40 ? "10-12" : "7"
- перші 300 символів кожного підрозділу (для конкретних висновків)
- conclusionsRequirements з методички (якщо є)
- textStructureHints (якщо є)
```

**Для ВИСНОВКІВ ДО РОЗДІЛУ додатково:**
```
- обсяг: 1 стор (~4-5 абзаців)
- перші 400 символів підрозділів цього розділу
- chapterConclusionRequirements з методички (якщо є)
```

**Для ОСНОВНИХ ПІДРОЗДІЛІВ додатково:**
```
- тип: theory/analysis/recommendations + опис що це означає
- вимоги методички для цього типу (theoryRequirements або analysisRequirements)
```

## СИСТЕМНИЙ ПРОМПТ (buildSYS)

Застосовується до всіх підрозділів. Містить:
- мова відповіді
- заборона markdown (#, **, *, -)
- заборона слів: аспект, важливий, особливий, значущий, ключовий, критичний, фундаментальний
- заборона тире "—" (заміна комою)
- стиль: короткі речення, активний стан, природні сполучники
- не повторювати назву підрозділу на початку

## ЩО ЗАРАЗ НЕ РЕАЛІЗОВАНО (потрібно додати)

1. **introPages / conclusionsPages не витягуються з методички** — промпт аналізу методички не має цих полів
2. **Коментар ділиться тільки на 2 поля** (planHints, writingHints) — треба 4
3. **Немає buildWorkConfig()** — логіка розкидана по гілках A-E
4. **Мінімум висновків** — гілки A і E мають `max(1,...)`, решта `max(2,...)` — неузгоджено
